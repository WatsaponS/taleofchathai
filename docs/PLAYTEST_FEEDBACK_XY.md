  ที่ขอบแมพตรงตามสเปก
- **NPC ใช้สไปรต์ pixel trainer จริง + กล่องคุยสไตล์ Pokémon** ทำงานถูกต้อง (ชนแล้วคุย เดินอีกทีปิดกล่อง)

  uncharted + แถบ main route ล่างจอ ครบตามที่ออกแบบไว้ทุกอย่าง ดีกว่าของ ChaThaiTheCat ชัดเจน
- **มอนสเตอร์เป็น GIF ขยับได้จริงทั้งในไฟต์** ตามที่ขอไว้รอบก่อน และฝั่งผู้เล่นถูก flip ด้วย CSS
  (`.battle-player-sprite .mon-img { transform: scaleX(-1) }`) ให้หันเข้าหาศัตรูแล้ว — บทเรียนจาก
  feedback รอบ ChaThaiTheCat ถูกนำมาใช้

## 1. โจทย์หลัก: การเดินยังเป็น 4 ทิศแบบ FireRed — X/Y ต้องการ 8 ทิศ + wheel control

### 1.1 สภาพปัจจุบัน (จากโค้ดจริง)

- `KEY_DIRS` ใน `main.js` แมป **1 ปุ่ม → 1 ทิศ (4 ทิศเท่านั้น)** ไม่มีการจำสถานะปุ่มที่กดค้างอยู่
  (ไม่มี keyup tracking) — กดขึ้น+ขวาพร้อมกันได้แค่ทิศใดทิศหนึ่ง เดินเฉียงเป็นไปไม่ได้เลยในสถาปัตยกรรมนี้
- **`fieldStepLock` กลืน input**: ทดสอบยิง keydown 40 ครั้งรัว ๆ ตัวละครขยับแค่ 1 ก้าว — input ที่มาระหว่าง
  ก้าว (150ms) ถูกทิ้งหมด ไม่มี buffer แถมการกดค้างพึ่ง OS key-repeat ซึ่งมี delay ช่วงแรก (~500ms) ก่อน
  จะรัว ทำให้ก้าวแรกกับก้าวที่สองมีจังหวะสะดุดเสมอ (เกม Pokémon จริงกดค้างแล้วไหลลื่นทันที)
- D-pad บนจอเป็น **ปุ่ม 4 ปุ่มแยกกัน** — กดทีละทิศ ไม่ใช่ wheel/analog ตามที่ผู้เล่นอยากได้

### 1.2 ข้อเท็จจริงจาก X/Y ที่ช่วยกำหนดสโคป

Pokémon X/Y มีการเดิน 2 ระดับ: (1) **เดินปกติด้วย Circle Pad = 8 ทิศแบบ grid** (เฉียงได้ แต่ยัง snap
กับตาราง) และ (2) **โรลเลอร์สเก็ต = free movement จริง** (ไม่ snap grid เลย) — แปลว่า "ให้เหมือน X/Y"
ไม่จำเป็นต้องรื้อระบบ grid ทิ้ง: **ทำ 8 ทิศบน grid เดิมก็คือการเดินปกติของ X/Y แล้ว** ส่วน free movement
เต็มรูปแบบเก็บเป็น stretch (เทียบเท่าสเก็ต) ได้

### 1.3 ข้อเสนอ implement (เรียงจากคุ้มสุด)

**ขั้นที่ 1 — Key-state tracking + step timer ของตัวเอง (ต้องทำก่อน เป็นฐานของทุกอย่าง)**:
เปลี่ยนจาก "keydown → เดินทันที" เป็น เก็บ `Set` ของปุ่มที่กดค้าง (`keydown` เพิ่ม / `keyup` ลบ) แล้วมี
game tick ของตัวเอง (~150ms interval หรือ rAF) อ่านจาก Set นั้นทุกงวด:

```js
const held = new Set();
document.addEventListener('keydown', e => { if (KEY_DIRS[e.key]) { held.add(e.key); e.preventDefault(); } });
document.addEventListener('keyup', e => held.delete(e.key));
setInterval(() => {
  if (state.screen !== 'field' || fieldStepLock) return;
  let dx = 0, dy = 0;
  for (const k of held) { dx += KEY_DIRS[k][0]; dy += KEY_DIRS[k][1]; }
  dx = Math.sign(dx); dy = Math.sign(dy);
  if (dx || dy) performFieldMove(dx, dy);
}, 150);
```

แก้ 2 ปัญหาพร้อมกัน: กดค้างเดินลื่นไม่ติด OS repeat delay และเปิดทางให้ (dx,dy) เป็นเฉียงได้ทันที

**ขั้นที่ 2 — รองรับก้าวเฉียงใน `moveFieldPlayer`**: ตอนนี้รับ (dx,dy) ที่แกนเดียวเท่านั้น ต้องเพิ่มกติกา
เฉียงแบบมาตรฐาน:
- ก้าวเฉียง (dx≠0 && dy≠0) เดินได้ก็ต่อเมื่อ **ช่องปลายทาง + ช่องข้างทั้งสองแกน** เดินได้ทั้งหมด (กันมุด
  ทะลุมุมกำแพง — corner cutting)
- ถ้าเฉียงไม่ผ่านแต่แกนใดแกนหนึ่งผ่าน ให้ **slide** ไปตามแกนที่ผ่าน (ฟีลลื่นแบบ analog ไม่ใช่ชนแล้วหยุดนิ่ง)
- `FACING_OF_DELTA` ต้องเพิ่ม 4 ทิศเฉียง (หรือ fallback เป็นแกนหลักของทิศเฉียงก็พอสำหรับ facing เพราะ
  สไปรต์มีแค่ 4 ทิศ) — เส้นสายตา trainer (`trainerSightHit`) เป็นแนวแกนตรงอยู่แล้ว ไม่ต้องแก้
- Encounter ต่อก้าวคิดเหมือนเดิม (ก้าวเฉียง = 1 ก้าว 1 โรล ไม่ใช่ 2)

**ขั้นที่ 3 — Wheel control (virtual analog stick) แทน D-pad 4 ปุ่ม**: ตอบคำขอตรง ๆ — แทนที่
`.dpad` ด้วย joystick เสมือนวงกลมเดียว (ฐานวงกลม + หัว nub ที่ลากได้) จับ `pointerdown/move/up` บนวงฐาน
คำนวณมุมจากจุดศูนย์กลาง → แปลงเป็น 8 ทิศ (แบ่งวงเป็น 8 ช่วง ช่วงละ 45°) แล้ว feed เข้า loop เดียวกับ
ขั้นที่ 1 (แทน `held` set ด้วยทิศจาก joystick) — โค้ด ~40-60 บรรทัด ไม่ต้องใช้ library:

```
       ▲
    ↖     ↗        มุมจาก atan2(dy,dx) → ปัด 45° → (dx,dy) ∈ {-1,0,1}²
  ◀   ●───o  ▶     deadzone ตรงกลาง ~25% ของรัศมี (แตะเฉย ๆ ไม่เดิน)
    ↙     ↘        ปล่อยนิ้ว = หยุดทันที
       ▼
```

**ขั้นที่ 4 (stretch — "โหมดสเก็ต" ของจริง)**: free movement แบบ pixel-based (ตำแหน่งเป็น float px ไม่
snap tile, collision เช็คเป็นกล่องกับ tile grid) — ได้ฟีล X/Y เต็มขั้นแต่ต้องรื้อ warp/NPC-bump/encounter
ให้เช็คแบบ box-overlap แทน tile เดียว งานใหญ่กว่าขั้น 1-3 รวมกัน แนะนำทำหลังขั้น 1-3 stable แล้วเท่านั้น
(หรือทำเป็น toggle "Run/Skate" ให้ผู้เล่นเลือก แบบเดียวกับที่ X/Y แยกเดินกับสเก็ตจริง ๆ)

## 2. การหันหน้า — เจอปัญหาจริง 3 จุด (เชื่อมกับ feedback รอบก่อนเรื่องหันหน้าหาศัตรู)

### 2.1 GIF เดินถูกใช้เป็นท่ายืนในไฟต์ → มอนหันหนีศัตรูเป็นพัก ๆ

เจอตอนเล่นจริง: เฟรมแรกของไฟต์ Street Pup (ฝั่งผู้เล่น) หันซ้าย = **หันหนีศัตรู** ทั้งที่ CSS flip ทำงาน
ถูกต้อง — สาเหตุไม่ใช่ CSS แต่คือ **`-move.gif` เป็นแอนิเมชันเดิน/เพซไปมา ตัวมอนขยับตำแหน่ง+กลับทิศเอง
ภายในแคนวาสของ GIF** (ยืนยันด้วยการวัด center-of-mass ต่อเฟรมของ street-pup-move.gif: มวลพิกเซลสวิง
ซ้าย↔ขวาเป็นวงจร) — CSS flip ภายนอกช่วยไม่ได้เพราะข้างใน GIF หมุนตัวเอง

X/Y โมเดลในไฟต์**หันหน้าเข้าหาคู่ต่อสู้ตลอดเวลา**ด้วยท่า idle เฉพาะ — ทางแก้เรียงตามถูก→แพง:
1. **ใช้ `-move-sheet.png` + JSON เลือกเฉพาะเฟรมที่หันทิศเดียวกัน** มาเล่นเป็น idle loop ในไฟต์ (มี
   frame-level data อยู่แล้ว ไม่ต้องผลิต asset ใหม่ — เขียน CSS `background-position` step animation
   เฉพาะ subset เฟรมที่คัดไว้) — วิธีนี้ต้องคัดเฟรมต่อสปีชีส์ครั้งเดียว (สคริปต์วัด center-of-mass แบบ
   ที่ใช้ตรวจครั้งนี้ automate ได้เลย)
2. ใช้ภาพนิ่ง `slug.png` ในไฟต์ + CSS idle bob เบา ๆ (แบบที่ ChaThaiTheCat ทำ) — ง่ายสุดแต่เสียความ
   เป็น GIF ขยับที่เพิ่งได้มา
3. ขอ asset ใหม่: idle GIF แยกจาก move GIF ต่อสปีชีส์ (แพงสุด 180+ ตัว — ไม่แนะนำตอนนี้)

**หมายเหตุเพิ่ม**: ยังไม่มีอะไรการันตีว่าทั้ง 180 สปีชีส์ GIF ต้นฉบับหันทิศเดียวกันหมด (Field Mouse ที่เจอ
หันถูกฝั่งพอดี แต่เป็นการสุ่มดวง) — ควรรันสคริปต์เช็ค native facing ทั้ง 180 ตัวแล้วเก็บเป็น metadata
ต่อสปีชีส์ (`faces: 'left'|'right'`) ให้ UI ตัดสินใจ flip ต่อตัว ไม่ใช่ flip ฝั่งเดียวแบบเหมารวม

### 2.2 ทิศ lunge ตอนโจมตีกลับด้าน — ผู้โจมตีถอยหลังแทนที่จะพุ่งใส่

จากโค้ด `style.css`: `lungeToEnemy` เคลื่อน `translate(-30px, 6px)` = **ซ้าย-ล่าง** แต่ศัตรูอยู่**ขวา-บน**
(player platform ล่างซ้าย / enemy บนขวา) และ `lungeToPlayer` เคลื่อน `translate(30px, -6px)` = ขวา-บน
ทั้งที่ผู้เล่นอยู่ล่างซ้าย — **ทั้งสองอันพุ่งหนีเป้าหมายแทนที่จะพุ่งเข้าหา** อ่านแล้วเหมือนผู้โจมตีสะดุ้งถอย
เอง สลับเครื่องหมายทั้งสอง keyframe ก็จบ (บั๊กเล็กแก้ 2 บรรทัด แต่เห็นทุกเทิร์นการต่อสู้)

### 2.3 ตัวละครผู้เล่นในฉากเดิน มี flip แค่ซ้าย-ขวา ไม่มีหันขึ้น/ลง

`.field-player-sprite` มีแค่ `face-left` (scaleX flip) — เดินขึ้น/ลงตัวละครหน้าตรงตลอด (ยืนคุยกับ NPC
ที่อยู่ด้านบนแต่ตัวละครหันหน้าเข้ากล้อง) — walk frames 8 เฟรมต่อ trainer ที่มีอยู่ (`player-trainers/walk/`)
ควรเช็คว่าครอบคลุม 4 ทิศไหม (ตามที่ `GAME_DESIGN.md` §1 ตั้งคำถามไว้แล้วเรื่อง mapping 8 เฟรม → 4 ทิศ
ยังไม่มีใครยืนยัน) ถ้าเฟรมมีครบให้ผูก `f.facing` → ชุดเฟรมที่ถูกต้อง ถ้าไม่ครบ ขั้นต่ำคือทำภาพ back-view
เพิ่ม 1 ท่าต่อ trainer (5 ภาพ) เพราะเดินขึ้น (หันหลังให้กล้อง) คือทิศที่ผู้เล่นเห็นบ่อยที่สุดในเกมแนวนี้

## 3. Comment อื่น ๆ (เล็กกว่า เรียงตามผลกระทบ)

1. **ไม่มี EXP bar ใน HUD ฝั่งผู้เล่น** — X/Y (และทุกภาคหลัง Gen 2) มีแถบ EXP สีฟ้าใต้แถบ HP ฝั่งผู้เล่น
   ทำให้รู้ว่าอีกไกลแค่ไหนจะเลเวลอัพโดยไม่ต้องเปิดเมนู — HUD ปัจจุบันมีช่องว่างพอ เพิ่มได้เลย
2. **Encounter ตัดเข้าไฟต์ค่อนข้างฉับ** — X/Y มี transition หมุน/แฟลชก่อนเข้าไฟต์เสมอ ถ้ามี flash effect
   อยู่แล้ว (เห็น `pendingEncounter` แยก state ไว้ให้เล่น FX ได้) ลองขยายเป็น flash ขาว-ดำ 2-3 จังหวะ +
   เสียง sting สั้น ๆ จะได้ฟีลใกล้เคียงขึ้นมาก
3. **ก้าวแรกหลัง encounter/dialogue จบ ต้องกดใหม่** — ถ้าถือปุ่มค้างอยู่ระหว่างที่กล่องคุยเปิด ปุ่มค้าง
   นั้นไม่ resume การเดินหลังกล่องปิด (ต้องปล่อยแล้วกดใหม่) — แก้ฟรีถ้าทำ key-state tracking ตามข้อ 1.3
   ขั้นที่ 1 เพราะ loop จะอ่านปุ่มที่ยังค้างอยู่เองในงวดถัดไป
4. **`-move.gif` ตัวเดียวใช้ทั้งเดิน+ไฟต์+Dex** — พอแก้ข้อ 2.1 (เฟรม idle สำหรับไฟต์) แนะนำใช้หลักเดียว
   กันกับ MonsterDex ด้วย (ใน Dex การ์ดควรนิ่งหรือ idle สม่ำเสมอ ไม่ใช่เพซไปมาจนอ่านยาก)
5. **Region Map ควรกดหมุดแล้วมี tooltip ชื่อโซน+สถานะก่อน travel จริง** — ตอนนี้ (จากโค้ด) กดหมุดคือ
   travel ทันที ถ้าเพิ่ม confirm ชั้นเดียว (ชื่อ+ระดับเลเวลแนะนำ+ปุ่ม Go) จะลด mis-tap บนมือถือด้วย —
   สอดคล้องกับที่เกมนี้มีโฟลเดอร์ `android/` แปลว่าตั้งใจลงมือถือจริง ยิ่งควรคิดเรื่อง touch ergonomics
   (ข้อนี้กับ wheel control ข้อ 1.3 คือชุดเดียวกัน: "เกมนี้เล่นบนจอสัมผัสเป็นหลัก")
